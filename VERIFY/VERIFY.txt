SAIL Verification Guide

SAIL v5.9.0 Evidence Integrity Runtime

Purpose

This directory contains the minimum verification artifacts required to validate the SAIL reference implementation and its five reference applications.

Verification focuses on observable structural behavior, accurate evidence classification, deterministic identities, certificates, guarded transitions, replayable outcomes, mutation sensitivity, and consistent realization states.

The objective is to determine whether documented guarantees remain true under repetition, variation, blocked conditions, structural change, and adversarial testing.

Primary Verification Invariant

same structure -> same evidence -> same certificate -> same replayable outcome

Supporting Invariants

same structural payload -> same structural identity

separate runtime record -> separate operational identity

operational identity and timestamps -> excluded from structural certificates

same accepted lifecycle transition repeated -> same certificate and unchanged transition timestamp

changed structural input -> affected downstream certificates reset

downstream certification allowed iff required upstream structures are certified

environment-dependent expression -> ABSTAIN

same rejected expression + changed browser environment -> same rejection state + same certificate

==================================================
STRUCTURAL AND OPERATIONAL IDENTITY
==================================================

SAIL v5.9.0 reference applications distinguish structural identity from operational identity.

Structural identity describes the deterministic structure being certified.

Operational identity describes a particular runtime record or operational instance.

Verification must confirm:

same structure -> same structural ID

separate runtime record -> separate operational record ID

record IDs do not alter structural certificates

timestamps do not alter structural certificates

operational metadata remains available for traceability

==================================================
LIFECYCLE GUARD VALIDATION
==================================================

Verification must test both successful and blocked transitions.

Examples include:

- realization blocked before approval
- approval blocked before trace certification
- trace blocked when policy is absent
- recovery blocked before required escalation
- verification blocked before recovery
- federation replay blocked before network resolution
- federation audit blocked before replay
- incomplete structures blocked from downstream certification

A blocked transition must not issue the downstream certificate.

==================================================
IDEMPOTENCY VALIDATION
==================================================

Accepted lifecycle transitions must be idempotent.

Verification must confirm:

- repeated approval preserves approval certificate and timestamp
- repeated realization preserves realization certificate and timestamp
- repeated escalation preserves escalation certificate and timestamp
- repeated recovery preserves recovery certificate and timestamp
- repeated verification preserves verification certificate and timestamp
- repeated accepted transition does not duplicate structural outcomes

==================================================
STRUCTURAL CHANGE VALIDATION
==================================================

When structural inputs change, certificates derived from the prior structure must not remain valid.

Verification must confirm:

- changed requirement structure resets approval and realization
- changed policy resets downstream requirement governance
- changed trace evidence resets downstream approval and realization
- changed incident structure resets escalation, recovery, verification, and realization
- changed control structure resets dependent compliance artifacts
- changed federation configuration invalidates dependent network-resolution artifacts

==================================================
REPLAY VALIDATION
==================================================

Replay validation must confirm:

same certified structure -> same replay result

same certificate chain -> same chain certificate

same federation structure -> same federated outcome

Replay must fail closed when required certificates or upstream states are absent.

==================================================
FEDERATION VALIDATION
==================================================

Federation verification must confirm:

quorum belongs to network configuration, not node identity

same node structure -> same node certificate

quorum certified iff certified node count >= configured quorum

consensus certified iff quorum met AND network identity aligned AND scope aligned

federation certified iff network resolved AND quorum certified AND consensus certified

federated replay allowed iff federation certified

federated audit allowed iff federation certified AND replay certified

Required failure scenarios include:

- quorum failure
- network identity mismatch
- scope mismatch
- pending-node exclusion
- unresolved federation replay
- audit before replay

==================================================
REALIZATION VALIDATION
==================================================

Realization verification must confirm that measurable outcome evidence is structurally connected to the certified lifecycle.

Core Realization Invariant

`realization certified iff required upstream structures, execution evidence, measurement, and verification are complete`

Realization certificates must not be issued when required preconditions are absent.

==================================================
FALSIFICATION TARGETS
==================================================

Attempt to produce:

same structure -> different structural identity

same structure -> different certificate

same visible input -> different replay result

separate runtime record -> same operational record identity

timestamp change -> changed structural certificate

missing policy -> policy certificate issued

missing measurement -> trace certificate issued

unapproved requirement -> realization certificate issued

required escalation absent -> recovery certificate issued

recovery absent -> verification certificate issued

quorum not met -> federation certified

network mismatch -> consensus certified

scope mismatch -> consensus certified

replay before federation resolution -> replay certificate issued

audit before replay -> federation audit certificate issued

unresolved content -> certified pack state

changed structure -> prior downstream certificate preserved

behavioral failure -> evidence validation passes

controlled source mutation undetected -> mutation validation passes

browser-global or environment-dependent expression -> resolved or certified result

undeclared identifier, property traversal, or constructor access -> resolved expression

browser-environment change -> changed rejection state or rejection certificate

environment-isolation check count differs from 7

environment-isolation failure count differs from 0

uncertified compliance evaluation -> COMPLIANT

certified FAIL or EXCEPTION evaluation -> COMPLIANT

synthetic conformance execution count -> represented as unique input, distinct scenario, or independent behavioral count

Successful falsification attempts are valid verification outcomes.

Reproducible violations should be used to:

- correct the implementation
- strengthen validation
- narrow documented guarantees
- improve release verification

==================================================
CERTIFICATION SCOPE
==================================================

SAIL certification denotes internal structural conformance and behavioral validation within the supplied reference implementation.

It is not third-party, regulatory, safety, security, compliance, governmental, or standards certification.

==================================================
FINAL VERIFICATION PRINCIPLE
==================================================

Verification is not confirmation by assumption.

Verification is the independent attempt to determine whether documented structural guarantees and evidence classifications remain true under repetition, variation, blocked conditions, structural change, mutation, and adversarial testing.
