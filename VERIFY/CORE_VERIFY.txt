SAIL Core Runtime Verification

Runtime

SAIL v5.9.0 Evidence Integrity Runtime

Purpose

This document defines the minimum verification procedure for the SAIL core runtime.

Core verification focuses on observable structural behavior, evidence classification, independently specified behavioral validation, mutation sensitivity, deterministic certification, replay stability, count integrity, and certified layer coverage.

The core runtime must be verified independently before validating the reference applications.

Evidence Classification Principle

The 347400 figure represents total synthetic structural conformance execution volume over bounded structural case-template sets.

The conformance audit validates deterministic hashing, canonicalization, cloning, replay, certificate consistency, component integration, structural self-consistency, and expected execution-count integrity.

Some conformance-audit families repeatedly cycle through their available structural templates until their configured execution totals are reached. This repetition tests deterministic stability and replay consistency under repeated execution.

The 347400 executions must not be represented as:

- 347400 unique input structures
- 347400 distinct scenarios
- 347400 unique structural templates
- 347400 independent behavioral specifications

Evidence interpretation:

347400 = synthetic structural conformance execution volume

347400 != unique structural template count

347400 != independent behavioral specification count

The independently specified behavioral evidence supplied with the release consists of 96 literal behavioral specifications.

==================================================
ENVIRONMENT-ISOLATION VALIDATION
==================================================

The external runtime behavioral test must execute seven environment-isolation checks.

These checks verify that mathematical and logical expressions cannot depend on browser-global state or undeclared execution-environment values.

Expected Results

environment_isolation_checks: 7

environment_isolation_failures: 0

Required Rejection Cases

* document and DOM property access
* window property access
* location property access
* object-property traversal
* constructor access
* undeclared identifiers
* nondeterministic environment functions

Validated Inputs

sqrt(document.title.length)

sqrt(window.innerWidth)

sqrt(location.href.length)

sqrt(document.body.children.length)

sqrt.constructor(1)

sqrt(unknown_identifier)

Math.random()

Expected Result for Every Input

status: ABSTAIN

certificate_stable: true

Environment-Isolation Invariants

`environment-dependent expression -> ABSTAIN`

`undeclared identifier -> ABSTAIN`

`property access -> ABSTAIN`

`constructor access -> ABSTAIN`

`same rejected expression + changed browser environment -> same rejection state + same certificate`

Validation Count Distinction

The seven environment-isolation checks are supplemental external runtime checks.

They are not part of the 96 independently specified behavioral specifications.

`behavioral specification count = 96`

`environment-isolation check count = 7`

`96 behavioral specifications + 7 supplemental environment-isolation checks`

Environment-Isolation Validity Rule

`environment isolation valid iff all 7 checks execute AND every environment-dependent expression abstains AND every rejection certificate remains stable`

==================================================
PRIMARY DETERMINISM INVARIANT
==================================================

`same structure -> same evidence -> same certificate -> same replayable outcome`

Supporting invariants:

`same visible input -> same certified result`

`same structural payload -> same structural identity`

`same certificate inputs -> same certificate`

`same certified structure -> same replay result`

==================================================
DETERMINISM BOUNDARY
==================================================

Certificate inputs must be derived from visible structural fields only.

The following must not influence structural certificate generation:

- random identifiers
- timestamps
- interface state
- operational record identifiers
- hidden mutable state
- external state absent from the structural input

Structural certificates must remain stable when only operational metadata changes.

Operational distinction:

`structural identity != operational record identity`

==================================================
STRUCTURAL IDENTITY VERIFICATION
==================================================

Verification must confirm:

- equivalent structures produce equivalent structural identities
- equivalent structures produce equivalent certificates
- separate runtime records may have separate operational identities
- operational record IDs do not alter structural certificates
- timestamps do not alter structural certificates
- export delivery mode does not alter structural certificates

Identity invariant:

`same structural payload -> same structural identity`

Operational invariant:

`separate runtime record -> separate operational identity`

==================================================
REPLAY VERIFICATION
==================================================

Replay verification must confirm:

`same certified structure -> same replay result`

Replay must preserve:

- structural identity
- evidence identity
- certificate identity
- resolved state
- realization state
- federation state where applicable

Replay must not depend upon:

- execution timestamp
- interface selection
- operational record identity
- export time
- download or view mode

==================================================
FAIL-CLOSED VERIFICATION
==================================================

The core runtime must fail closed when required structure is absent or inconsistent.

Verification should confirm that:

- incomplete structure does not produce a successful certificate
- contradictory structure does not resolve as certified
- missing evidence does not produce verified realization
- insufficient federation participation does not produce certified quorum
- network disagreement does not produce consensus
- unresolved federation does not produce replay certification
- invalid downstream states do not bypass upstream guards
- behavioral failures prevent evidence validation from passing
- undetected controlled source mutations prevent mutation validation from passing

Fail-Closed Principle

`certification denied unless all required structural conditions are satisfied`

==================================================
FALSIFICATION TARGETS
==================================================

Attempt to produce:

- same structure -> different structural identity
- same structure -> different evidence
- same structure -> different certificate
- same certified structure -> different replay result
- timestamp change -> changed structural certificate
- operational record change -> changed structural certificate
- missing evidence -> verified realization
- invalid structure -> certified result
- quorum not met -> federation certified
- network mismatch -> consensus certified
- scope mismatch -> consensus certified
- unresolved federation -> replay certificate
- behavioral failure -> behaviorAudit allPass true
- behavioral failure -> evidenceValidation allPass true
- undetected controlled mutation -> evidenceValidation allPass true
- synthetic conformance execution count represented as unique inputs, distinct scenarios, unique templates, or independent behavioral cases
- failed audit cases -> allPass true
- incorrect count -> release accepted
- fewer than 37 layers -> core certified

A reproducible falsification result is a valid verification outcome.

It must be used to:

- correct the implementation
- strengthen validation
- narrow the documented guarantee
- update the release status
